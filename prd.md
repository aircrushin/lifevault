

# 产品需求文档 (PRD): LifeVault - 个人健康数据资产化 Web 钱包

| 文档版本 | V1.0 |
| --- | --- |
| **产品名称** | LifeVault |
| **产品类型** | Web3 / HealthTech / DataFi |
| **核心定位** | 个人健康数据的“瑞士银行”——存储、加密、生息 |
| **目标用户** | 量化自我者 (Bio-hackers)、慢性病患者、Web3 早期采用者 |

---

## 1. 产品概述 (Product Overview)

### 1.1 背景与痛点

目前，用户的体检报告和基因数据分散在各个医院和检测机构，产生了两大问题：

1. **数据孤岛：** 用户无法统一管理自己的健康全貌。
2. **价值流失：** 医药公司利用用户数据获利，但作为数据源头的用户却从未获得分红。

### 1.2 产品愿景

LifeVault 致力于打造一个 Web3 风格的隐私保险库，通过**“数据即资产 (Data as an Asset)”**的理念，让用户拥有数据主权，并通过授权数据给科研机构赚取被动收益。

### 1.3 核心价值主张

* **Security (安全):** 只有用户持有解密私钥。
* **Liquidity (流动性):** 将静态的体检报告转化为可产生现金流的资产。
* **Insight (洞察):** 将生物学术语转化为金融评分。

---

## 2. 用户角色 (User Personas)

1. **“数据矿工” (Data Staker):** 25-40岁，科技敏感，做过基因检测，希望通过现有数据获得 Token 或现金奖励。
2. **特定病患 (The Candidate):** 拥有罕见病或特定慢性病史，医药研发公司急需此类数据，属于“高价值资产”持有者。

---

## 3. 功能需求详情 (Functional Requirements)

### 3.1 模块一：Data Vault (数据金库)

**功能描述：** 用户的资产存储中心，支持多格式健康数据上传与价值锚定。

| ID | 功能点 | 详细描述 | 优先级 |
| --- | --- | --- | --- |
| **F1.1** | **多格式上传** | 支持拖拽上传 PDF (体检报告)、JPG (化验单)、.vcf/.fastq (基因测序原始数据)。 | P0 |
| **F1.2** | **端到端加密** | 文件在浏览器端进行本地加密 (Local Encryption) 后再上传至 IPFS/Arweave 或云存储，确保平台方也无法查看原始内容。 | P0 |
| **F1.3** | **OCR 数据解析** | 自动识别 PDF/图片中的关键指标（如血糖、胆固醇、白细胞计数）并结构化存入数据库。 | P1 |
| **F1.4** | **资产估值引擎** | 根据数据完整度、稀缺性（如罕见基因突变）和市场需求，实时计算数据的“估值” (Data Valuation)。显示格式：`$150.00 / Month`。 | P1 |

> **交互细节：** 上传过程应模仿 DeFi 的“存款”体验，进度条显示“正在铸造数据资产...”，完成后显示“资产已入库”。

### 3.2 模块二：Health Scorecard (健康金融分)

**功能描述：** 将生物信息转化为类似“信用分”的直观指标，用于量化用户的健康资产质量。

| ID | 功能点 | 详细描述 | 优先级 |
| --- | --- | --- | --- |
| **F2.1** | **评分模型** | 综合基因风险、当前体检指标、生活习惯，计算出 300-900 分的“Health Credit Score”。 | P0 |
| **F2.2** | **雷达图可视化** | 从五个维度展示健康：心血管、免疫力、代谢、遗传风险、抗衰老能力。 | P1 |
| **F2.3** | **价值洞察 (Insights)** | 提示用户：“您的低密度脂蛋白数据对心血管药物研发具有高价值”。 | P2 |

### 3.3 模块三：Consent Management (授权与收益)

**功能描述：** 类似于“理财超市”，用户在此决定将数据“质押”给谁。

| ID | 功能点 | 详细描述 | 优先级 |
| --- | --- | --- | --- |
| **F3.1** | **需求方列表** | 展示寻求数据的机构（如：辉瑞、某大学实验室）。每个卡片显示：收购目的、所需数据类型、报价 (APY 或 一次性)。 | P0 |
| **F3.2** | **一键授权开关** | 简洁的 Toggle Switch (开关)。开启即代表签署智能合约/电子协议，允许该机构在特定期限内访问脱敏数据。 | P0 |
| **F3.3** | **收益看板** | 实时显示 `Total Earnings` (总收益)、`Pending Payout` (待结算)。支持法币或 Token 提现。 | P1 |
| **F3.4** | **访问日志** | 透明展示谁在何时调用了数据（例如：“2024-05-20 14:00, XYZ Lab 用于糖尿病模型训练”）。 | P2 |

## 4. 业务逻辑与算法 (Logic & Algorithms)

### 4.1 数据估值逻辑 (Data Valuation Logic)

* ** (基础价值):** 只要是真实人体数据即拥有的底价。
* ** (完整度):** 数据维度越全（基因+体检+穿戴设备），系数越高 (0.5 - 2.0)。
* ** (稀缺性):** 若用户携带罕见突变或特殊病史，系数极高。
* ** (市场需求):** 当前医药市场对该类数据的热度（动态浮动）。

### 4.2 隐私保护流程

1. **用户上传** -> 2. **本地生成密钥对** -> 3. **公钥加密数据** -> 4. **存储密文**。
2. **授权发生时** -> 6. **零知识证明 (ZKP)** 验证数据符合要求（不泄露原始数据） 或 **代理重加密** 给买方。

---

## 5. 非功能性需求 (Non-Functional Requirements)

### 5.1 安全性 (Security) & 合规

* **HIPAA / GDPR:** 必须符合医疗数据处理规范。
* **Wallet Connect:** 支持 MetaMask, Coinbase Wallet 等 Web3 钱包登录，以此作为身份验证 (DID)。
* **Audit Trail:** 所有数据访问记录必须上链或存入不可篡改的日志库。

### 5.2 性能 (Performance)

* **大文件处理:** 基因文件 (.vcf) 可能高达数百 MB，需支持断点续传和前端流式解析。
* **响应速度:** 估值计算需在 3 秒内完成，给用户即时反馈。

## 6. UI/UX 关键页面 (Wireframe Concepts)

1. **Dashboard (首页):**
* 顶部：总资产估值 (Total Data Equity)。
* 中部：健康雷达图 + 健康金融分。
* 底部：待处理的“数据变现”机会（推荐的授权项目）。


2. **Upload Center (上传页):**
* 极简的拖拽区。
* 动画效果：文件被粉碎成二进制流并加上“锁”的动画，暗示加密过程。


3. **Marketplace (授权页):**
* 列表式布局，右侧为显眼的绿色“开启授权”开关和对应的金额收益。



---

## 7. 数据埋点与指标 (Metrics)

* **北极星指标 (North Star Metric):** **TVL (Total Value Locked)** - 平台托管的健康数据总估值。
* **核心指标:**
* 用户上传率 (注册用户中完成至少一份报告上传的比例)。
* 授权转化率 (上传用户中开启至少一个授权开关的比例)。
* 平均用户收益 (ARPU)。
